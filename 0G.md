# Reckon Protocol — 0G Integration

## Project Overview

Reckon is an autonomous EBBO (Execution Best Bid Offer) enforcement protocol for UniswapX. AI-powered **challenger agents** — each represented by an ERC-7857 iNFT on 0G Galileo — monitor solver fills, detect price violations against on-chain benchmarks, and submit slashing challenges without human intervention.

0G is not an add-on. It provides the **identity layer** (iNFTs), the **intelligence layer** (encrypted brain blobs on 0G Storage), the **reasoning layer** (LLM inference via 0G Compute), and the **coordination layer** (distributed KV for multi-agent deconfliction). Removing 0G would require rewriting every component of the agent stack.

## 0G Features Used

### 1. ERC-7857 iNFTs — Agent Identity & Encrypted Intelligence

Each challenger agent is minted as an **ERC-7857 Intelligent NFT** on 0G Galileo via the `ChallengerNFT` contract. The iNFT standard (from `0g-agent-nft`) provides:

- **On-chain identity**: Each agent has a unique `tokenId` that identifies it across all protocol interactions — challenge submissions, claim coordination, reputation tracking.
- **Encrypted brain blobs**: The agent's configuration (model preferences, EBBO thresholds, Ed25519 keys, performance history) is AES-256-GCM encrypted and stored on 0G Storage. The `tokenURI` points to the Merkle root hash of this encrypted blob.
- **Owner-controlled decryption**: The encryption key is derived via PBKDF2 from the iNFT owner's wallet signature. Only the owner can decrypt the brain, and the agent boots by unsealing it at startup.
- **Brain rotation**: The `updateBrain()` function allows the owner to push new model configs or updated thresholds without re-minting — the brain evolves while the identity stays stable.
- **Sealed-key transfers**: ERC-7857's `iTransferFrom` emits `PublishedSealedKey` events, enabling secure intelligence transfer between owners via on-chain proof verification.

**Proof that intelligence/memory is embedded in the iNFT:**

The brain blob stored at each `tokenURI` contains:
```typescript
interface BrainBlob {
  axl_ed25519_secret: string;         // Ed25519 key for inter-agent signing
  ebbo_threshold_prefs: {              // Challenge decision parameters
    minSlash: string;                  // Minimum slash to justify a challenge
    maxBondPct: number;                // Maximum bond risk per challenge
  };
  kh_api_key: string;                 // Per-agent KeeperHub credential
  model_config: { model: string };    // 0G Compute model preference
  performance_history: SlashRecord[];  // Historical slash outcomes (learning memory)
}
```

This is not static metadata — it is the agent's **operational intelligence**. The `performance_history` array records past slash outcomes, enabling the agent to learn from its decisions. The `ebbo_threshold_prefs` are tunable per-agent, meaning different agents can develop different risk profiles. The `axl_ed25519_secret` is the agent's cryptographic identity for signing claim messages in the coordination layer.

### 2. 0G Storage — Brain Blobs & Fill Batch Archival

0G Storage serves two purposes in Reckon:

#### a) Brain Blob Storage (iNFT Intelligence)

When an agent is provisioned, its encrypted brain blob is uploaded to 0G Storage via `@0gfoundation/0g-ts-sdk`. The upload returns a **Merkle root hash** that becomes the iNFT's `tokenURI` on-chain. At boot, the agent:

1. Reads `tokenURI` from ChallengerNFT on Galileo → gets the Merkle root hash
2. Downloads the brain blob from 0G Storage via the indexer (with Merkle proof verification: `withProof=true`)
3. Decrypts using the owner's PBKDF2-derived key
4. Initializes with the decrypted model config, thresholds, and Ed25519 identity

Brain root hashes for provisioned agents:
- Sentinel (#0): `0xf9b98c78606c7ca007e8c8cefd6c7d7906b9a400af0223659e814c89442312e4`
- Warden (#2): `0x3f4662c8558908ba12d5a4abbef2397bd351d7132c51fb9d3d5166576ff8dbd8`

#### b) Fill Batch Archival (Data Availability)

The indexer batches fill records (50 per batch or every 600 seconds) and uploads them to 0G Storage as JSON Lines files. Each batch's Merkle root is anchored on-chain via `FillRegistry.anchorBatch()`, creating a verifiable link between on-chain events and the full off-chain data. This enables:

- Any party to independently verify fill data by downloading from 0G Storage and checking the Merkle root
- Historical auditability of all solver fills without bloating on-chain storage
- Replay and dispute resolution using cryptographically anchored data

### 3. 0G Storage KV — Multi-Agent Claim Coordination

When multiple challenger agents detect the same slashable fill, only one should submit the challenge. Reckon uses **0G Storage KV** as the **durable coordination layer** for claim deconfliction:

- A single protocol-wide KV stream (`CLAIM_STATE_STREAM_ID = keccak256("reckon/claim_state/v1")`) stores claim state
- Each fill's `orderHash` maps to a claim record: `{agentTokenId, claimedAt, deadline}`
- The coordination protocol is:
  1. **Check** — Read the KV stream to see if the fill is already claimed
  2. **Acquire** — Write a claim to KV after winning the AXL gossip backoff window
  3. **Verify** — Read-after-write to confirm no race condition (KV is the source of truth)
  4. **Release** — Zero out the claim if the agent decides not to challenge

The flow contract used for KV writes: `0x22E03a6A89B950F1c82ec5e74F8eCa321a105296` (0G Galileo)

### 4. 0G Compute — AI-Powered Fill Triage & Explanation

Every fill passes through an **LLM triage step** before the computationally expensive EBBO benchmark check. The agent uses the **0G Compute Router API** (OpenAI-compatible) to run inference on 0G's decentralized compute network:

#### a) Suspicion Triage (pre-filter)

The agent sends each fill's details (input/output amounts, tolerance, filler identity) to an LLM and asks it to score the fill from 0.0 (normal) to 1.0 (suspicious). Fills scoring below 0.3 are skipped entirely, saving unnecessary RPC calls and gas estimation.

#### b) Slash Explanation (post-decision)

After a challenge decision is made, the agent generates a **natural-language explanation** of why the fill was slashed — suitable for a dashboard or audit log. This makes the protocol's enforcement decisions interpretable by humans.

## How Agents Communicate and Coordinate (Swarm Behavior)

Reckon agents form a **swarm** where each agent independently monitors fills but must coordinate to avoid duplicate challenges. The coordination uses a two-layer protocol:

### Layer 1: AXL Gossip (Fast, Ephemeral)

Each agent runs alongside a [Gensyn AXL](https://github.com/gensyn-ai/axl) node. When an agent decides a fill is worth challenging, it:

1. **Signs** a claim message with its Ed25519 key (from the brain blob)
2. **Broadcasts** the signed claim to all peers via AXL's encrypted P2P mesh
3. **Listens** for competing claims during a backoff window (30 seconds)
4. **Yields** if a peer's claim has an earlier timestamp (or lower tokenId as tiebreaker)

The Ed25519 keys used for signing are stored inside each agent's encrypted brain blob on 0G Storage — making the iNFT the root of the agent's cryptographic identity.

### Layer 2: 0G Storage KV (Slow, Durable)

After surviving the AXL gossip round, the agent writes its claim to 0G KV as the **source of truth**. A read-after-write verification ensures no race condition occurred. If two agents write simultaneously, the read-back reveals the conflict and the loser yields.

This two-layer design means:
- **Happy path** (no contention): AXL resolves in ~30 seconds, no KV write needed beyond the winner's claim
- **Contention path**: KV provides a globally consistent tiebreaker even if AXL messages are lost
- **Recovery**: Expired claims (past deadline) are automatically treated as unclaimed
